# Design: project backlog markdown + completion tracking

**Status:** Approved (decisions locked) тАФ ships as 1.3.0
**Scope:** make the persisted backlog a visible, progress-tracked project doc

## Problem
1.2.0 made the extracted backlog deterministic + cached (`~/.cache/orchestrator/intake/*.json`),
so `--intent` is stable. But the cache is **invisible** (hidden dir, opaque JSON) and tracks
**no completion state** тАФ you can't see how much of a page is built.

## Locked decisions
1. **Location: both.** A canonical local `BACKLOG.md` (always written), plus the same file
   refreshed *into the target repo* on `--live` runs (so the feature PR carries current progress).
2. **Done = PR merged.** Status flips to done only when the feature PR actually merges
   (`sdlc complete`), not when the PR opens.
3. **JSON canonical + Markdown view.** The cache JSON stays the source of truth (exact specs,
   `--intent` resolution); `BACKLOG.md` is rendered from it. Hand-edits to the md are advisory.

## Status model
Per intent, stored in the cache file under a new `progress` map (`{intent_id: {...}}`), so it
survives `--refresh` (deterministic ids map cleanly):

| status | when | marker |
|---|---|---|
| `todo` | extracted, not started | `[ ]` |
| `in_progress` | `--live` opened a PR | `[~]` |
| `done` | PR merged (`sdlc complete`) | `[x]` |

Each entry also records `issue_key`, `pr_url`, and `updated`.

## Rendered markdown
```markdown
# Backlog тАФ confluence://3365404675
<!-- generated by orchestrator; edits are advisory -->
Progress: 3 / 27 done ┬╖ 1 in progress

- [x] intent-create-page-object тАФ Create Page Object  (AEO-39 ┬╖ PR #4)
- [~] intent-implement-html-fetching-in-crawler-layer тАФ HTML Fetching  (AEO-40 ┬╖ PR #5)
- [ ] intent-handle-redirects-in-crawler-layer тАФ Handle Redirects
```

## Components
- **`intake/cache.py`** тАФ add `load_progress` / `set_progress(source, intent_id, тАж)`; `analyze_cached`
  preserves existing progress when re-extracting (don't wipe done markers on `--refresh`).
- **`intake/backlog_doc.py`** тАФ pure `render_markdown(source, plan, progress) -> str` +
  `write_backlog(path, тАж)`; default local path `./BACKLOG.md` (or `$ORCHESTRATOR_BACKLOG_PATH`).
- **`sdlc/feature_runner.py`** тАФ after analyze, (re)write local `BACKLOG.md`. On `--live` after the
  PR opens: `set_progress(in_progress, issue_key, pr_url)`, then write `BACKLOG.md` *into the
  worktree* and include it in the commit/PR (the "both" location).
- **CLI**
  - `sdlc complete --pr тАж` тАФ on a confirmed merge, `set_progress(done)` + rewrite local `BACKLOG.md`.
  - new `orchestrator backlog --source тАж [--out BACKLOG.md]` тАФ render the current backlog + progress
    from the cache (read-only; handy to view or regenerate).

## Testing
- `render_markdown`: markers per status, progress summary, issue/PR annotations.
- `set_progress` round-trips through the cache file; `--refresh` preserves progress for surviving
  intents and drops vanished ones.
- feature_runner (stubbed): a `--live` run writes `BACKLOG.md` locally and into the worktree and
  marks the intent `in_progress`.

## Rollout
- Ships as 1.3.0. Backward compatible: a cache without `progress` reads as all-`todo`; `BACKLOG.md`
  generation is additive.
- Docs: USER_GUIDE тАФ note `BACKLOG.md` + the `backlog` command + the done-on-merge flow.
