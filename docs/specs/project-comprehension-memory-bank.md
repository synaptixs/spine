# Design: project comprehension — the memory bank (Phase 0 entrypoint)

**Status:** Phases 1 + 4 implemented (1.5.0 + 1.6.0). 1.5.0: `orchestrator/knowledge/`
(`understand.py`, `renderers.py`) + `orchestrator understand` CLI render structural files.
1.6.0: `knowledge/access.py` feeds the committed memory bank into codegen grounding
(`PKGCodegenGrounder`) + a `read_memory_bank` MCP tool. Remaining: Phase 2 (synthesized
brief/product-context prose) and `components.md` (needs the UI extractor).
**Goal:** a new first entrypoint that builds a durable, committed **project knowledge
base** — a markdown **memory bank** backed by the PKG knowledge graph — *before* intake
or codegen, so every downstream phase grounds on maintained project understanding.

## Reframe
Today the entrypoint is requirements-first (`ingest` / `sdlc feature`). This adds a
**knowledge-first Phase 0**: `orchestrator understand`. It comprehends the project
(source + docs + a little synthesized prose), persists it as `memory-bank/*.md` **in the
target repo**, and that becomes the foundation everything else reads.

> This generalizes the Supernova-beating thesis: not just design-system context, but
> whole-project context — code-derived, provenance-grounded, persisted, human- and
> agent-readable.

## Locked decisions
1. **Location:** `memory-bank/` committed **into the target repo** (like `BACKLOG.md`).
   Travels with the code, visible to the team + other agents/IDEs, version-controlled drift.
2. **Generation:** **rendered-from-PKG first.** Structural files are rendered
   deterministically from PKG facts (accurate, free, fast); an LLM synthesizes **only**
   the brief / product-context prose, cached and re-run only on `--refresh`.

## Two branches (same output shape)
| | **Brownfield** (existing code) | **Greenfield** (new/empty) |
|---|---|---|
| Detect | `pkg.load_or_extract` finds real code; `is_effectively_empty` (from `sdlc/layout.py`) is False | empty / no package |
| Build | **extract** PKG → render structural files; **synthesize** brief/context from README + facts + `pkg docs` reconciliation | **seed** brief/context from the intake brief/intents; scaffold (reuse 1.1.0); render whatever facts exist (often just the skeleton) |
| Note | reflects shipped reality | "greenfield — knowledge grows as features land" |

## The memory bank (`memory-bank/` in the target repo)
Each file headed `<!-- generated by `orchestrator understand`; rendered from PKG @ <sha>; edits advisory -->`.

| File | Source | Kind |
|---|---|---|
| `architecture.md` | PKG modules/types/`CONTAINS`, dir layering, `stats.py` call hotspots | **rendered** |
| `domain-model.md` | PKG `Entity`/`Field`/`REFERENCES` (+ data-layer `READS`/`WRITES` once emitted) | **rendered** |
| `components.md` | PKG `COMPONENT`/`PROP`/`RENDERS`/`USES_TOKEN` (from the UI extractor spec) | **rendered** *(empty until that extractor lands)* |
| `tech-context.md` | `catalog/profile.ProjectProfile.from_repo` (langs/framework/db/tests/task-type) | **rendered** |
| `conventions.md` | codegen convention digest (G8) + lint configs (ruff/eslint) | **rendered** |
| `glossary.md` | registry glossary + domain terms | **rendered** |
| `progress.md` | **the existing `BACKLOG.md` ledger (1.3.0)** — link/co-locate | reused |
| `project-brief.md` | what/why/goals | **synthesized** (cached) |
| `product-context.md` | domain, users, problem | **synthesized** (cached) |
| `active-context.md` | current focus; updated by feature runs | light/append |

## Architecture (new `orchestrator/knowledge/` package)
- `memory_bank.py` — manifest (which files, their kind), location resolution
  (`<repo>/memory-bank/`, override `$ORCHESTRATOR_MEMORY_BANK_DIR`), write/read.
- `renderers.py` — pure `facts/profile → markdown` per structural file (deterministic,
  no LLM). Mirrors the `backlog_doc.render_markdown` pattern.
- `synthesize.py` — LLM prose for brief/product-context; cached with a content hash so a
  refresh only re-runs when inputs changed. `temperature=0` (consistent with intake).
- `understand.py` — orchestrates: detect branch → extract/seed → render → synthesize →
  write the bank; returns a manifest summary.
- **CLI:** `orchestrator understand [PATH] [--brief FILE|--source <uri>] [--refresh]`.

## Canonical graph ↔ markdown view (reuse the 1.3.0 pattern)
Same split we shipped for the backlog:
- **PKG facts = canonical** (queryable, provenance, deterministic).
- **memory bank = the rendered + narrated view** — structural files regenerate from facts
  (always accurate); prose files are synthesized once and cached. Hand-edits are advisory;
  `--refresh` regenerates. No new canonical store invented.

## Downstream consumption
- **Codegen** keeps grounding on PKG (live query) — same facts — but `conventions.md` and
  `components.md`/`domain-model.md` become a stable, committed grounding source the
  `PKGCodegenGrounder` can prepend (esp. conventions, which are otherwise re-derived per run).
- **Intake** writes better specs/gap-analysis against `domain-model.md` + `glossary.md`.
- **MCP / external agents** — add a `read_memory_bank(section)` tool so Cursor/Claude Code
  consume the project knowledge directly (the distribution parity with authored tools).

## Freshness
- Brownfield: PKG cache is commit-keyed (`pkg/persistence.py`), so structural renders are
  cheap and always current; `understand --refresh` re-extracts + re-renders; prose re-runs
  only when its input hash changed.
- Greenfield: the bank grows as features land; `sdlc feature` updates `progress.md`
  (already) and can append `active-context.md`.
- CI check (optional, later): fail if `memory-bank/` structural files are stale vs HEAD —
  the "your context drifted" signal, measured against code.

## Phased plan
1. **`knowledge/` package + `understand` CLI + memory-bank manifest** + the **rendered**
   files that need only *today's* PKG + profile (`architecture`, `domain-model`,
   `tech-context`, `conventions`, `glossary`). Wire `progress.md` to the existing ledger.
   → **buildable now**, no dependency on the UI extractor.
2. **Synthesized prose** (`project-brief`, `product-context`) — greenfield from intents,
   brownfield from README + facts; cached.
3. **Branch polish** — greenfield seed path (scaffold + brief), brownfield drift via `pkg docs`.
4. **Downstream wiring** — feed `conventions.md`/`domain-model.md` into codegen grounding;
   `read_memory_bank` MCP tool.
5. **`components.md`** lights up when the PKG UI extractor (other spec) lands.
6. **Freshness CI check** + outcome feedback into the bank.

## Honest limits
- Synthesized prose can lag intent; it's cached + regenerated on `--refresh`, and clearly
  marked advisory.
- `components.md` is a stub until the UI extractor ships (Phase 1 of the PKG-deepening spec).
- Comprehension is structural + usage-grounded, not deep semantic intent understanding.

## First step
Phase 1 against the orchestrator's **own** repo (Python, rich PKG): run
`orchestrator understand .` and produce a real `memory-bank/` — `architecture.md` +
`domain-model.md` + `tech-context.md` + `conventions.md` rendered from the live PKG — then
point it at the AEO repo to show the greenfield→growing path. That output is the proof:
*a committed, code-true knowledge base built with no manual authoring.*
